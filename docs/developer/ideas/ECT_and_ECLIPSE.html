Introduction

Could/should ECLIPSE (http://www.eclipse.org/) be used as an IDE for ECT? If so, how?

Eclipse

"Eclipse is an open source community whose projects are focused on providing an extensible development platform and application frameworks for building software." (http://www.eclipse.org/)

Perhaps the most mature use of Eclipse is as a Java IDE, although it aspires to greater generality. However, achieving a reasonable conceptual fit to (something like) developing a Java application is probably essential for using Eclipse to be worthwhile. From previous experience with the NetBeans IDE it is important it is extremely hard to work outside or against - for example - the paradigm of folders and files which are compiled into a "product" form.

So... what are the main concepts/elements of Eclipse?

  • Project - a unit of development, corresponding to a file-system folder and contents, annotated with one or more "natures", which determine what Perspectives and facilities are available in it.
  • Perspective - a configuration of the Workbench UI oriented to a particular task, e.g. a particular combination of menu and toolbar actions, Editors and Viewers.
  • Editor - an interactive editor for a particular file type.
  • View - a specialised view onto a resource, synchronized through the UI selection model, often a file.
  • Resource - file, folder
  • (can't remember the term) Annotation(?) - of a resource, e.g. a to-do item, bug.
  • Action - a task thing can be explicitly performed, may be associated with Editor(s), View(s) or menu or toolbar Action Sets.
    • One class of actions it to run a wizard, which guides a user through a stereotyped process which is typically equivalent to a set of other coordinated UI operations, optionally plus additional custom operations (e.g. template code generation).

In addition, the Java Development Tools (JDT) part of Eclipse also defines:

  • Builder - for building/compiling projects
  • Lancher - for running built projects
  • Debugger - for monitoring and interacting with a launched application.

ECT

ECT files and applications

In terms of Java code ECT consists of:

  • Common run-time, including dataspace tuple wrappers, type coercion, ECT-specific component interfaces, the implementation of the EQUIP-based container, various other supporting facilities (e.g. web server implementations).
  • (Heavy-weight Java) Components, each (currently) as one or more source files, optional manifest, possible dependent JARs and native libraries.
  • Separate tools, including:
    • graphical editor
    • simpler display editor
    • configuration manager
    • physical configuration manager 
  • WebStart boot and installation manager

Other files which currently exist include:

  • Documentation (mainly HTML)
  • configuration manager XML-based transferable configuration files
  • container XML-based capability and component persistence files
  • component-specific persistence files (for components implementing the Persistable interface)
  • graph editor XML-based layout persistence files
  • console logs
  • dataspace logs and checkpoints
  • dataspace configuration file(s)
  • installation and container (webstart) configuration files
  • processing sketches and dependencies (e.g. font files, images) (separate)

ECT in use

The normal way of using ECT has been:

  1. Check out of CVS & build
  2. Develop any additional Java components within the same source tree and build
  3. Run using scripts - simple or webstart
  4. Assemble configuration interactively using the graph editor
  5. Voila!
